Clase pentru diagrame UML
Aplicatia Max Imob

Autor: Claudiu Ștefan

Rolul documentului

Acest document centralizeaza clasele si relatiile principale care pot fi folosite la realizarea diagramelor UML pentru aplicatia Max Imob. Aplicatia este dezvoltata ca platforma web pentru administrarea si publicarea ofertelor imobiliare, avand trei categorii de utilizatori: vizitator public, agent imobiliar si administrator.

Documentul poate fi utilizat ca anexa pentru lucrarea de licenta sau ca suport pentru sectiunile de proiectare si implementare. Clasele sunt prezentate la nivel conceptual, pornind de la structura reala a aplicatiei: tabelele PostgreSQL, serviciile TypeScript, validatoarele si rutele API definite in proiect.

1. Clase de domeniu

Clasele de domeniu descriu obiectele principale ale aplicatiei. Acestea corespund entitatilor persistente din baza de date si sunt folosite in fluxurile de autentificare, administrare agenti, administrare proprietati si transmitere mesaje.


1.1. User

Clasa User reprezinta contul intern al sistemului. In aplicatie, conturi au doar administratorii si agentii imobiliari. Vizitatorii publici nu au conturi si nu se autentifica.

Atribute:
- id: string
- email: string
- passwordHash: string
- role: UserRole
- name: string | null
- phone: string | null
- isActive: boolean
- createdAt: Date
- updatedAt: Date

Metode conceptuale:
- isAdmin(): boolean
- isAgent(): boolean
- canAuthenticate(): boolean

Observatii:
- role poate avea valorile admin sau agent.
- passwordHash contine parola stocata in forma criptata, nu parola in clar.
- isActive este folosit pentru activarea sau dezactivarea agentilor.
- Administratorul poate gestiona agenti, dar nu gestioneaza proprietati.


1.2. Agent

Clasa Agent este o specializare logica a clasei User. In baza de date, agentii sunt stocati in tabela users, diferentierea fiind facuta prin campul role = agent.

Atribute:
- id: string
- name: string
- email: string
- phone: string | null
- isActive: boolean
- createdAt: string
- updatedAt: string

Metode conceptuale:
- createProperty(input: CreatePropertyInput): Property
- updateProperty(propertyId: string, input: UpdatePropertyInput): Property
- deactivateProperty(propertyId: string): Property
- viewMessages(): Message[]

Observatii:
- Un agent poate avea mai multe proprietati.
- Un agent poate primi mai multe mesaje prin proprietatile publicate.
- Un agent poate modifica doar proprietatile proprii.


1.3. Admin

Clasa Admin este o specializare logica a clasei User. Administratorul are rol de administrare a agentilor si nu participa la fluxurile de gestionare a proprietatilor.

Atribute:
- id: string
- email: string
- role: UserRole
- name: string | null
- isActive: boolean

Metode conceptuale:
- listAgents(status: AgentStatusFilter): Agent[]
- createAgent(input: CreateAgentInput): Agent
- updateAgent(agentId: string, input: UpdateAgentInput): Agent
- deactivateAgent(agentId: string): Agent

Observatii:
- Administratorul are acces la zona /admin.
- Administratorul poate crea, modifica, activa si dezactiva agenti.
- Administratorul nu creeaza si nu editeaza proprietati.


1.4. Property

Clasa Property reprezinta o oferta imobiliara introdusa de un agent. Proprietatile pot fi afisate public doar daca sunt publicate si active.

Atribute:
- id: string
- agentId: string
- title: string
- description: string
- price: number
- city: string
- address: string
- propertyType: PropertyType
- status: PropertyStatus
- offerType: PropertyOffer
- bedrooms: number
- bathrooms: number
- area: number
- isActive: boolean
- images: PropertyImage[]
- createdAt: string
- updatedAt: string

Metode conceptuale:
- publish(): void
- markAsSold(): void
- markAsRented(): void
- activate(): void
- deactivate(): void
- addImage(image: PropertyImage): void

Observatii:
- propertyType poate avea valorile apartament, casa, teren sau comercial.
- status poate avea valorile ciorna, publicata, vanduta sau inchiriata.
- offerType poate avea valorile vanzare sau inchiriere.
- Fiecare proprietate apartine unui singur agent.
- Proprietatile publice trebuie sa aiba status = publicata si isActive = true.


1.5. PropertyImage

Clasa PropertyImage reprezinta o imagine atasata unei proprietati.

Atribute:
- id: string
- propertyId: string
- imageUrl: string
- fileName: string
- sortOrder: number
- createdAt: Date

Metode conceptuale:
- getDisplayOrder(): number

Observatii:
- O proprietate poate avea mai multe imagini.
- Stergerea unei proprietati determina stergerea imaginilor asociate prin relatia ON DELETE CASCADE.
- sortOrder stabileste ordinea de afisare a imaginilor.


1.6. Message

Clasa Message reprezinta mesajul trimis de un vizitator public catre agentul responsabil de o proprietate. In cerintele aplicatiei, aceasta entitate poate fi considerata si lead.

Atribute:
- id: string
- propertyId: string
- agentId: string
- contactName: string
- contactEmail: string
- contactPhone: string
- messageContent: string
- status: MessageStatus
- createdAt: string
- updatedAt: string

Metode conceptuale:
- markAsRead(): void
- archive(): void

Observatii:
- MessageStatus poate avea valorile unread, read sau archived.
- Fiecare mesaj apartine unei proprietati.
- Fiecare mesaj este asociat agentului care detine proprietatea.
- Vizitatorul public nu are cont; datele sale sunt salvate doar in mesaj.


2. Clase de autentificare si autorizare

Aceste clase descriu partea de autentificare pe baza de email, parola, JWT si cookie HTTP-only.


2.1. AuthSession

Clasa AuthSession reprezinta informatiile extrase din tokenul JWT al utilizatorului autentificat.

Atribute:
- userId: string
- role: UserRole

Metode conceptuale:
- isAdmin(): boolean
- isAgent(): boolean

Observatii:
- Sesiunea contine doar informatiile necesare autorizarii.
- Tokenul JWT contine userId si role.


2.2. AuthService

Clasa AuthService reprezinta serviciul responsabil de autentificare.

Atribute:
- cookieName: string
- jwtSecret: string

Metode:
- loginUser(input: LoginInput): LoginResult
- findUserByEmail(email: string): StoredUser | null
- createAuthToken(userId: string, role: UserRole): string
- invalidCredentials(): LoginResult

Responsabilitati:
- Validarea existentei utilizatorului.
- Verificarea starii active a contului.
- Compararea parolei primite cu parola criptata.
- Generarea tokenului JWT.
- Stabilirea redirectarii catre /admin sau /agent.


2.3. SessionService

Clasa SessionService reprezinta logica de citire si verificare a sesiunii din request.

Metode:
- getAuthSession(request: NextRequest): AuthSession | null
- requireAdmin(request: NextRequest): AuthSession
- requireAgent(request: NextRequest): AuthSession

Responsabilitati:
- Citeste cookie-ul auth_token.
- Verifica tokenul JWT folosind cheia secreta.
- Permite accesul doar rolurilor autorizate.
- Arunca AuthError pentru cereri neautentificate sau neautorizate.


2.4. AuthError

Clasa AuthError reprezinta o eroare de autentificare sau autorizare.

Atribute:
- message: string
- status: 401 | 403

Metode:
- constructor(message: string, status: 401 | 403)

Observatii:
- status = 401 indica lipsa autentificarii.
- status = 403 indica lipsa drepturilor pentru resursa ceruta.


3. Clase de servicii

Serviciile contin logica aplicatiei si comunica direct cu baza de date. In arhitectura proiectului, controllerele/rutele API raman simple, iar logica principala este mutata in servicii.


3.1. AgentService

Metode:
- listAgents(status: AgentStatusFilter): Promise<Agent[]>
- createAgent(input: CreateAgentInput): Promise<Agent>
- getAgentById(id: string): Promise<Agent | null>
- updateAgent(id: string, input: UpdateAgentInput): Promise<Agent | null>
- deactivateAgent(id: string): Promise<Agent | null>
- isUniqueViolation(error: unknown): boolean
- mapAgentRow(row: AgentRow): Agent

Responsabilitati:
- Listeaza agentii.
- Creeaza agenti noi cu parola criptata.
- Actualizeaza datele agentului.
- Dezactiveaza conturi de agent.
- Transforma randurile SQL in obiecte Agent.


3.2. PropertyService

Metode:
- listAgentProperties(agentId: string): Promise<Property[]>
- listPublicProperties(filters: PublicPropertyFilters): Promise<Property[]>
- createProperty(agentId: string, input: CreatePropertyInput, images: SavedPropertyImage[]): Promise<Property>
- getAgentPropertyById(agentId: string, propertyId: string): Promise<Property | null>
- getPublicPropertyById(propertyId: string): Promise<Property | null>
- updateProperty(agentId: string, propertyId: string, input: UpdatePropertyInput, images: SavedPropertyImage[]): Promise<Property | null>
- deactivateProperty(agentId: string, propertyId: string): Promise<Property | null>
- activateProperty(agentId: string, propertyId: string): Promise<Property | null>
- listImagesForProperty(propertyId: string): Promise<PropertyImage[]>
- insertPropertyImages(client: PoolClient, propertyId: string, images: SavedPropertyImage[], startIndex: number): Promise<void>
- mapPropertyRow(row: PropertyRow, images: PropertyImage[]): Property

Responsabilitati:
- Gestioneaza proprietatile agentilor.
- Filtreaza proprietatile publice.
- Insereaza proprietati si imagini in tranzactie.
- Actualizeaza datele proprietatii.
- Activeaza si dezactiveaza proprietati.
- Respecta regula ca agentul modifica doar proprietatile proprii.


3.3. MessageService

Metode:
- saveMessage(propertyId: string, agentId: string, input: CreateMessageInput): Promise<Message>
- getPropertyById(propertyId: string): Promise<PropertyReference | undefined>
- getMessagesByPropertyId(propertyId: string): Promise<Message[]>
- getMessagesByAgentId(agentId: string): Promise<Message[]>
- markMessageAsRead(messageId: string): Promise<Message>
- mapMessageRow(row: MessageRow): Message

Responsabilitati:
- Salveaza mesajele trimise prin formularul public.
- Asociaza mesajul cu proprietatea si agentul corect.
- Listeaza mesajele pentru o proprietate.
- Listeaza mesajele pentru agentul autentificat.
- Marcheaza mesajele ca citite.


3.4. DatabasePool

Clasa DatabasePool reprezinta mecanismul centralizat de conectare la PostgreSQL.

Atribute:
- connectionString: string
- pgPool: Pool

Metode:
- getPool(): Pool

Responsabilitati:
- Creeaza conexiunea la baza de date.
- Reutilizeaza conexiunea in mediul de dezvoltare.
- Verifica existenta variabilei de mediu DATABASE_URL.


4. Clase de validare

Validatoarele verifica datele primite de la client inainte de apelarea serviciilor.


4.1. LoginValidator

Metode:
- validateLoginInput(input: unknown): LoginValidationResult

Reguli:
- email trebuie sa fie string valid.
- password trebuie sa fie string.
- parola trebuie sa aiba minimum 8 caractere.
- emailul este normalizat prin trim si lowercase.


4.2. AgentValidator

Metode:
- validateCreateAgentInput(input: unknown): ValidationResult<CreateAgentInput>
- validateUpdateAgentInput(input: unknown): ValidationResult<UpdateAgentInput>
- validateAgentStatusFilter(status: string | null): AgentStatusFilter | null

Reguli:
- name: 2-100 caractere.
- email: format valid.
- phone: optional, maximum 30 caractere.
- password: minimum 8 caractere.
- isActive: boolean.


4.3. PropertyValidator

Metode:
- validateCreatePropertyInput(input: unknown): ValidationResult<CreatePropertyInput>
- validateUpdatePropertyInput(input: unknown): ValidationResult<UpdatePropertyInput>
- validatePublicPropertyFilters(input: unknown): ValidationResult<PublicPropertyFilters>

Reguli:
- title: 3-150 caractere.
- description: obligatoriu, maximum 2000 caractere.
- price: numar mai mare sau egal cu 0.
- city: 2-100 caractere.
- address: 5-200 caractere.
- propertyType: apartament, casa, teren sau comercial.
- status: ciorna, publicata, vanduta sau inchiriata.
- offerType: vanzare sau inchiriere.
- bedrooms: numar intreg mai mare sau egal cu 0.
- bathrooms: numar intreg mai mare sau egal cu 0.
- area: numar mai mare decat 0.


4.4. MessageValidator

Metode:
- validateCreateMessageInput(input: unknown): ValidationResult<CreateMessageInput>

Reguli:
- contactName: 2-100 caractere.
- contactEmail: format valid.
- contactPhone: 10-20 caractere si caractere permise pentru numar de telefon.
- messageContent: 10-5000 caractere.


5. Clase de tip controller / route handler

In implementarea Next.js App Router, controllerele sunt reprezentate de fisiere route.ts. Ele primesc cererea HTTP, valideaza datele, verifica sesiunea unde este cazul, apeleaza serviciile si returneaza raspunsuri JSON.


5.1. AuthLoginRoute

Endpoint:
- POST /api/auth/login

Metode:
- POST(request: NextRequest): Promise<NextResponse>

Responsabilitati:
- Citeste emailul si parola.
- Apeleaza LoginValidator.
- Apeleaza AuthService.
- Seteaza cookie-ul HTTP-only cu tokenul JWT.
- Returneaza redirectarea potrivita pentru rol.


5.2. AgentsRoute

Endpoint:
- GET /api/agents
- POST /api/agents

Metode:
- GET(request: NextRequest): Promise<NextResponse>
- POST(request: NextRequest): Promise<NextResponse>

Responsabilitati:
- Permite acces doar administratorului.
- Listeaza agentii.
- Creeaza agenti noi.
- Gestioneaza erorile de email duplicat.


5.3. AgentByIdRoute

Endpoint:
- GET /api/agents/[id]
- PATCH /api/agents/[id]
- DELETE /api/agents/[id]

Metode:
- GET(request: NextRequest, context): Promise<NextResponse>
- PATCH(request: NextRequest, context): Promise<NextResponse>
- DELETE(request: NextRequest, context): Promise<NextResponse>

Responsabilitati:
- Permite acces doar administratorului.
- Returneaza detaliile unui agent.
- Actualizeaza datele agentului.
- Dezactiveaza agentul.


5.4. PropertiesRoute

Endpoint:
- GET /api/properties
- POST /api/properties

Metode:
- GET(request: NextRequest): Promise<NextResponse>
- POST(request: NextRequest): Promise<NextResponse>

Responsabilitati:
- Pentru GET public, returneaza proprietatile publicate si active.
- Pentru POST, permite doar agentilor autentificati sa creeze proprietati.
- Valideaza filtrele publice si datele proprietatii.
- Proceseaza imaginile atasate, daca exista.


5.5. PropertyByIdRoute

Endpoint:
- GET /api/properties/[id]
- PATCH /api/properties/[id]
- DELETE /api/properties/[id]

Metode:
- GET(request: NextRequest, context): Promise<NextResponse>
- PATCH(request: NextRequest, context): Promise<NextResponse>
- DELETE(request: NextRequest, context): Promise<NextResponse>

Responsabilitati:
- Pentru GET public, returneaza o proprietate publicata si activa.
- Pentru PATCH si DELETE, permite acces doar agentului care detine proprietatea.
- Actualizeaza sau dezactiveaza proprietatea.


5.6. PropertyMessagesRoute

Endpoint:
- POST /api/properties/[id]/messages

Metode:
- POST(request: NextRequest, context): Promise<NextResponse>

Responsabilitati:
- Permite vizitatorilor publici sa trimita mesaje.
- Valideaza datele formularului de contact.
- Verifica existenta proprietatii.
- Salveaza mesajul pentru agentul responsabil.


5.7. AgentMessagesRoute

Endpoint:
- GET /api/agent/messages

Metode:
- GET(request: NextRequest): Promise<NextResponse>

Responsabilitati:
- Permite acces doar agentilor autentificati.
- Returneaza mesajele primite de agent.

7. Relatii intre clase

Relatii principale:
- Admin mosteneste conceptual User.
- Agent mosteneste conceptual User.
- Agent are zero sau mai multe Property.
- Property apartine unui singur Agent.
- Property are zero sau mai multe PropertyImage.
- PropertyImage apartine unei singure Property.
- Property are zero sau mai multe Message.
- Message apartine unei singure Property.
- Message apartine unui singur Agent.
- AuthService foloseste User pentru autentificare.
- SessionService produce AuthSession.
- AgentService gestioneaza Agent.
- PropertyService gestioneaza Property si PropertyImage.
- MessageService gestioneaza Message.
- Route handlers folosesc validatoare, servicii si SessionService.

Multiplicitati:
- Agent 1 ---- 0..* Property
- Property 1 ---- 0..* PropertyImage
- Property 1 ---- 0..* Message
- Agent 1 ---- 0..* Message
- User 1 ---- 0..1 AuthSession

11. Observatii pentru integrarea in lucrarea de licenta

Pentru lucrarea scrisa, aceste diagrame pot fi introduse in Capitolul 4 - Proiectarea sistemului, in sectiunea dedicata proiectarii orientate pe obiecte si structurii interne a aplicatiei. Diagrama claselor de domeniu este cea mai importanta pentru explicarea modelului de date, deoarece arata legatura dintre agenti, proprietati, imagini si mesaje.

Diagrama serviciilor poate fi folosita pentru a evidentia separarea responsabilitatilor: rutele API primesc cererile, validatoarele verifica datele, serviciile executa logica aplicatiei, iar DatabasePool gestioneaza accesul la PostgreSQL. Aceasta separare sustine cerintele de mentenabilitate si face aplicatia mai usor de explicat in cadrul sustinerii lucrarii.

Diagrama validatoarelor si rutelor API poate fi folosita pentru a prezenta fluxul cererilor HTTP si modul in care aplicatia respecta regulile de securitate. Zonele administrative folosesc SessionService pentru autorizare, iar zona publica permite doar operatii fara autentificare, precum vizualizarea proprietatilor si trimiterea mesajelor.


12. Diagrama cazurilor de utilizare

Diagrama cazurilor de utilizare descrie functionalitatile principale ale aplicatiei din perspectiva actorilor. Pentru aplicatia Max Imob, actorii importanti sunt vizitatorul public, agentul imobiliar si administratorul. Aceasta diagrama este utila in Capitolul 2 - Analiza cerintelor, deoarece face legatura dintre cerintele functionale si rolurile reale din sistem.

Vizitatorul public poate accesa partea publica a aplicatiei fara autentificare. El poate consulta lista de proprietati, poate filtra proprietatile, poate deschide pagina de detaliu a unei proprietati si poate trimite un mesaj catre agentul responsabil. Aceasta alegere respecta regula de business conform careia utilizatorii publici nu au conturi in sistem.

Agentul imobiliar foloseste aplicatia dupa autentificare. El poate administra propriile proprietati, poate adauga imagini, poate modifica statusul unei proprietati si poate consulta mesajele primite de la vizitatori. Restrictia principala este ca agentul poate lucra numai cu datele care ii apartin.

Administratorul are acces la zona de administrare si poate gestiona conturile agentilor. El poate crea agenti, poate modifica datele lor si poate activa sau dezactiva conturi. Administratorul nu are rol de agent si nu gestioneaza direct proprietati.

13. Diagrama de secventa pentru autentificare

Fluxul de autentificare este folosit doar de administratori si agenti. Vizitatorii publici nu folosesc acest flux. Diagrama de secventa evidentiaza pasii prin care formularul de login comunica cu ruta API, validatorul, serviciul de autentificare si baza de date.

Din punct de vedere al securitatii, parola introdusa de utilizator nu este comparata direct cu o parola in clar. Backend-ul cauta utilizatorul dupa email, verifica daca acesta este activ, compara parola cu hash-ul salvat folosind bcrypt si apoi genereaza un token JWT. Tokenul este trimis catre browser printr-un cookie HTTP-only, reducand riscul accesarii lui prin cod JavaScript din client.

participant "Pagina Login" as LoginPage
participant "POST /api/auth/login" as LoginRoute
participant "LoginValidator" as Validator
participant "AuthService" as AuthService
database "PostgreSQL" as DB
participant "JWT + Cookie HTTP-only" as Cookie

LoginPage -> LoginRoute : trimite cerere POST

alt date invalide
  Validator --> LoginRoute : eroare validare
  LoginRoute --> LoginPage : raspuns 400
else date valide
  Validator --> LoginRoute : input normalizat
  LoginRoute -> AuthService : loginUser(input)
  AuthService -> DB : cauta utilizator dupa email
  DB --> AuthService : user sau null

  alt credentiale incorecte sau cont inactiv
    AuthService --> LoginRoute : rezultat invalid
    LoginRoute --> LoginPage : raspuns 401
  else credentiale valide

  end

14. Diagrama de secventa pentru trimiterea unui mesaj public

Trimiterea mesajelor este unul dintre fluxurile esentiale ale aplicatiei, deoarece transforma vizitatorul public intr-un potential client pentru agent. Acest flux nu necesita autentificare, dar necesita validarea datelor introduse in formularul de contact.

Sistemul verifica mai intai datele formularului: nume, email, telefon si continutul mesajului. Apoi verifica existenta proprietatii si identifica agentul care o detine. Mesajul este salvat in baza de date impreuna cu propertyId si agentId, ceea ce permite agentului sa vada ulterior mesajele asociate proprietatilor sale.
