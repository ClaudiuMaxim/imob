ANEXE

Anexa 1. Structura proiectului

Aplicatia Max Imob este organizata in directoare care separa responsabilitatile principale:

- app - contine paginile, componentele si rutele API ale aplicatiei Next.js;
- app/api - contine endpoint-urile pentru autentificare, agenti, proprietati si mesaje;
- app/admin - contine zona administratorului;
- app/agent - contine zona agentului;
- app/properties - contine zona publica pentru proprietati;
- lib - contine serviciile, validarile si logica reutilizabila;
- lib/auth - contine autentificarea si gestionarea sesiunii;
- lib/agents - contine serviciile si validarile pentru agenti;
- lib/properties - contine serviciile pentru proprietati, imagini si mesaje;
- lib/db - contine configurarea conexiunii la baza de date;
- db/migrations - contine migratiile SQL;
- docs/licenta - contine documentatia lucrarii.

Anexa 2. Exemple de migratii SQL

Exemple de fisiere de migrare utilizate in proiect:

- 001_create_users.sql - creeaza tabela users si rolurile admin/agent;
- 003_create_properties.sql - creeaza tabela properties si tipurile pentru proprietati;
- 004_create_property_images.sql - creeaza tabela property_images;
- 005_create_messages.sql - creeaza tabela messages;
- 006_add_property_offer_type.sql - adauga tipul de oferta vanzare/inchiriere.

Aceste migratii definesc structura bazei de date si relatiile dintre entitati. Ele permit reproducerea bazei de date intr-un mediu nou si documenteaza evolutia sistemului.

Anexa 3. Exemple de rute API

Rute API principale:

- POST /api/auth/login - autentificarea administratorului sau agentului;
- GET /api/agents - listarea agentilor, disponibila administratorului;
- POST /api/agents - crearea unui agent, disponibila administratorului;
- PUT /api/agents/[id] - actualizarea unui agent;
- DELETE /api/agents/[id] - dezactivarea unui agent;
- GET /api/properties - listarea proprietatilor publice sau a proprietatilor agentului;
- POST /api/properties - crearea unei proprietati de catre agent;
- GET /api/properties/[id] - obtinerea detaliilor unei proprietati;
- PUT /api/properties/[id] - actualizarea unei proprietati de catre agent;
- PATCH /api/properties/[id] - activarea unei proprietati;
- DELETE /api/properties/[id] - dezactivarea unei proprietati;
- POST /api/properties/[id]/messages - trimiterea unui mesaj public;
- GET /api/agent/messages - listarea mesajelor primite de agent.

Anexa 5. Scenarii de testare

Scenarii principale folosite la validarea aplicatiei:

1. Autentificarea administratorului cu date corecte.
2. Autentificarea agentului cu date corecte.
3. Respingerea autentificarii cu parola gresita.
4. Crearea unui agent de catre administrator.
5. Dezactivarea unui agent si verificarea imposibilitatii de autentificare.
6. Crearea unei proprietati de catre agent.
7. Editarea unei proprietati proprii.
8. Dezactivarea si reactivarea unei proprietati.
9. Afisarea proprietatilor publicate in zona publica.
10. Filtrarea proprietatilor dupa oras, tip si oferta.
11. Trimiterea unui mesaj prin formularul de contact.
12. Afisarea mesajului in panoul agentului responsabil.
13. Verificarea faptului ca un agent nu vede mesajele altui agent.
14. Respingerea accesului unui agent la rutele administratorului.
15. Respingerea datelor invalide in formularele principale.
